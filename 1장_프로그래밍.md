# 내용정리
## 프로그래밍이란?
- 대부분의 문제(요구사항)는 복잡하며 명확하지 않을 수도 있다. 
- 따라서 요구사항을 명확히 이해하는 것이 우선되어야 하며 복잡함을 단순하게 분해하고 자료를 정리하고 구분해야 하며 순서에 맞게 행위를 배열해야 한다. 
- 문제해결 방안을 고려할 때 컴퓨터의 입장에서 문제를 바라봐야 한다.
- 여기에는 논리적, 수ㄱ적 사고가 필요하며, 해ㄹ 과제를 작은 단위로 분해하고 패턴화해 추출하며, 프그래밍 내에 사용될 모든 개념은 평가 가능하도록 정의해야 한다.

## 구문과 의미
- 프로그래밍은 요구사항의 집합을 분석해 적절한 자료구조와 함수의 집합으로 변환한 후, 그 흐름을 제어하는 것이다.

## 실습문제
🧩 문제 1. 요구사항 → 알고리즘 → 코드
요구사항: “사용자의 나이에 따라 입장 가능 여부를 출력하는 프로그램을 만들어라.”
18세 이상이면 “입장 가능”
18세 미만이면 “입장 불가”
```js
// TODO: 아래의 로직을 작성하세요
// 1. 입력값 age를 받아서
// 2. 조건에 따라 메시지를 출력하세요.
```

🧩 문제 2. 구문과 의미의 차이 실험
아래 코드 중 문법적으로는 올바르지만, 의미적으로 잘못된 예를 찾아 설명해보세요.
``` js
const x = "10";
const y = 5;
console.log(x + y);  // ?
```
→ 문법상 오류는 없지만, 의미상 계산 로직이 맞지 않습니다.
이처럼 “구문적으로 올바르지만 의미가 잘못된 코드”를 찾는 것은 프로그래밍의 핵심 사고 훈련이에요.

💬 3️⃣ 프론트엔드 면접 포인트
질문 1: “프로그래밍에서 구문과 의미의 차이는 실무에서 어떻게 나타날까요?”
핵심 포인트:
구문은 “코드가 실행될 수 있는가”의 문제
의미는 “코드가 올바르게 작동하는가”의 문제
예시:
onClick={handleClick()} vs onClick={handleClick}
구문은 모두 맞지만, 의미는 완전히 다름 (첫 번째는 즉시 실행, 두 번째는 핸들러 전달)
질문 2: “복잡한 문제를 단순하게 분해한다는 것은 프론트엔드에서 어떤 상황에 해당하나요?”
예시:
복잡한 UI 요구사항을 컴포넌트 단위로 나누는 것
Button, Input, Modal 등 작은 단위로 나누어 재조합하는 사고
관련 개념: 컴포넌트화(Componentization), 단일 책임 원칙(SRP), UI 상태(State) 관리
